# Grok Build Desktop 整体项目复审（2026-07-15）

## 结论

0.1.1 已针对推理强度、输入焦点、滚动、界面密度、子 Agent 状态和桌面交付进行专项修复，并完成一次覆盖 Renderer、Electron 主进程、ACP、账号保险库、更新器与脚本的整体复审。

## 已关闭问题

### P0：安全与数据完整性

- Electron 仅信任打包后的本地 Renderer；开发态仅允许回环地址，并拦截非预期导航、重定向、子框架 IPC 和非 HTTP(S) 外链。
- Plan Gate 拒绝命令链、管道、重定向、换行、子表达式及带写入能力的参数，避免只读模式绕过。
- JSON 元数据写入改为串行队列、唯一临时文件和成功后更新缓存，避免并发写损坏。
- OAuth/API Key 切换加入事务文件、启动残留恢复、超时、进程树终止与失败回滚；应用退出会取消仍在进行的登录。
- 日志、导出日志及 CLI 更新历史会再次脱敏 API Key、JWT、刷新令牌、授权头和 URL 查询令牌。

### P1：核心交互与生命周期

- 推理强度在 Grok CLI `0.2.101` 上点击即切换，并等待真实 `model_changed` 通知确认；旧 CLI 才走重启恢复兼容路径。
- Renderer 不再使用阻塞式原生确认/输入框；设置、文件选择、会话操作和自定义对话框关闭后恢复输入焦点。
- 发送后强制显示新消息；处于底部时持续跟随流式回复，用户主动上滚时不抢位置，并提供返回底部按钮。
- 会话恢复时保留“等待历史消息后滚到底部”的意图，避免本地状态与 Zustand 更新顺序导致的首次滚动竞态。
- 普通 `turn_completed` 不再被识别成子 Agent；子 Agent 和后台任务按明确生命周期更新并在回合结束时收敛。
- 进程启动、退出、取消及终端销毁均清理进程树；崩溃会从实时会话池移除，取消期间不会提前显示空闲。
- 未识别 ACP 请求返回 method-not-found，不再用空对象伪成功；自动批准无允许项时安全拒绝。

### P2：响应速度与可维护性

- 流式事件按动画帧合并，消息与 Markdown 卡片使用 memo；Monaco Diff、Mermaid 和 Shiki 按需加载。
- CLI 更新检查移到首屏初始化之后；会话打开和工作区刷新增加竞态保护。
- 文字缩放和布局密度拆分：文字保持可读，密度单独控制侧栏、顶栏、会话行和编辑器留白。
- 更新器合并并发更新请求，超时后清理整个进程树，并保留主错误与恢复错误；PowerShell 更新脚本异步排空输出，默认不终止无关 Grok 进程。

## 当前建议配置

- 文字大小：`100%`
- 界面密度：`紧凑`

该组合保持截图中 100% 的可读字级，同时接近原来 70% 缩放时的留白和信息密度。若仍觉得拥挤，优先改为“均衡”密度，而不是缩小文字。

## 已知但可接受的剩余项

- Grok CLI 冷启动/恢复会话仍通常需要约 2–3 秒；这是独立 CLI 进程初始化成本，已避免在主界面线程上增加额外等待。
- 推理强度热切换依赖 Grok 私有扩展；未来 CLI 若更改协议会自动退回重启恢复，但需要继续维护兼容矩阵。
- 终端卡片在界面中保留最近 40 KiB 并明确显示“输出已截断”，防止极端输出拖垮 Renderer；因此不是无限量内嵌显示。
- 首版仍是 Windows 本地 `win-unpacked` 交付，不包含正式安装器和跨平台构建。
- 自动化已覆盖主进程、ACP 和状态归并；真实 Electron 交互采用可见窗口验收，后续可补充稳定的端到端回归套件。
